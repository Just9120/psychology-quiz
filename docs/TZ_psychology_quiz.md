# ТЗ: psychology-quiz (актуализировано после merge UX v2)

## 1. Назначение проекта
- Проект — Telegram-бот викторины по психологии.
- Продукт ориентирован на учебную практику и самопроверку по темам психологии.
- Текущий продуктовый контур: только Module 1.

## 2. Текущий продуктовый scope
- В runtime поддерживается только Module 1.
- Активные категории Module 1:
  - Введение в профессию
  - Общая психология
  - Физиология человека
  - Физиология ВНД
  - Психофизиология
- Режим работы бота: long polling.
- Web UI отсутствует.
- Webhook отсутствует.
- RAG/внешняя runtime-генерация вопросов отсутствуют.

## 3. Архитектурный принцип данных
- Source of truth для question bank: JSON-файлы в `content/questions/module1/`.
- SQLite не является источником истины; SQLite — runtime/storage layer.
- Обновление runtime-слоя выполняется через `scripts/seed_questions.py`.
- Официальная цепочка обновления контента:
  1. изменить JSON в репозитории;
  2. оформить PR;
  3. выполнить merge;
  4. запустить seed;
  5. получить актуализированную runtime SQLite.
- Ручное редактирование SQLite как способ обновления контента запрещено.

## 4. Структура question bank
- Основная директория Module 1: `content/questions/module1/`.
- Категории хранятся отдельными JSON-файлами.
- Базовая схема элемента вопроса:
  - `id`
  - `category`
  - `source_ref`
  - `difficulty`
  - `status`
  - `question`
  - `options`
  - `correct_option_index`
  - `explanation`
- В SQLite загружаются только записи со статусом `approved`.

## 5. UX сценарий викторины (текущее состояние)
Поддерживаются отдельные runtime-режимы:
- `Конкретная тема`
- `Микс из выбранных тем`
- `Все темы`

Потоки:
- `Конкретная тема` → выбор одной категории → выбор количества вопросов → выбор сложности.
- `Микс из выбранных тем` → мультивыбор категорий → `Готово` / `Сбросить` → выбор количества вопросов → выбор сложности.
- `Все темы` → выбор количества вопросов → выбор сложности.

Повторяемость выбранного микса:
- режим `Микс из выбранных тем` является отдельным runtime-flow;
- выбранный набор тем сохраняется в рамках пользовательской сессии;
- действие `Пройти еще раз` повторяет тот же selected mix.

## 6. Вспомогательный UX
- Используется persistent reply keyboard в private chat.
- Меню можно скрыть через `🙈 Скрыть меню`.
- После скрытия меню доступно восстановление через inline-кнопку `👁 Показать меню`.
- Дополнительный fallback для восстановления интерфейса — `/start`.
- Поддерживаются режимы чтения:
  - `Обычный`
  - `Бионическое чтение`

## 7. Контентные правила и source priority
- Контент question bank должен быть на русском языке.
- Вопросы без source grounding добавлять нельзя.
- Интернет не используется как первичный источник для наполнения question bank.
- Приоритет источников для нового контента:
  1. папка `Психология` с исходными лекциями, транскрибациями, презентациями и практиками;
  2. текущий question bank в репозитории;
  3. Obsidian/пересобранная knowledge-base как вторичный синтетический слой.
- Практико-ориентированные вопросы встраиваются в профильные категории, а не выделяются в отдельную runtime-категорию.

## 8. Операционный workflow
- Code-only deploy выполняется, когда меняется только логика приложения без изменений question bank.
- Если меняется JSON-контент, обязателен seed через `scripts/seed_questions.py`.
- Допускается пакетное сидирование после серии merge (по операционному решению), но source of truth всегда остается в Git JSON.
- Официальная последовательность синхронизации: JSON в репозитории → merge → seed → обновление runtime SQLite.

## 9. Ограничения и анти-паттерны
- Нельзя рассматривать Module 2 как текущий продуктовый scope.
- Нельзя обновлять контент через прямые правки SQLite.
- Нельзя добавлять вопросы без внятного `source_ref` и source grounding.
- Нельзя смешивать в question bank нерелевантный PM-layer.
- README является компактным operational overview; детальные нормы фиксируются в этом ТЗ.
